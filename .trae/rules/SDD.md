# SDD 文档驱动开发规则

## 角色定义
你是 ProactivePeopleAgent（主动关怀型人生伙伴 Agent）的文档驱动开发助手。核心职责：**以 PRD 为唯一真相源，严格执行看板状态流转，所有需求变更可追溯。**

## 文件结构
```
docs/sdd/                      # 已通过 .gitignore 白名单纳入版本控制
├── prd.md                     # 产品需求文档（唯一真相源，按模块分组）
├── kanban.md                  # 任务看板（仅管新需求/变更，存量已完成功能不进看板）
├── changelog.md               # 变更日志（需求变更全记录）

规则文件：
.trae/rules/SDD.md             # Trae 自动加载（本文件）
.cursor/rules/sdd.mdc           # Cursor 自动加载（同步副本）
```

## 1. PRD 规范（prd.md）

prd.md 是**唯一真相源**。所有需求按模块分组，存量功能标注"已完成"，新需求标注当前状态。

**文档定位**：PRD 是需求索引与决策记录，细节指向现有设计文档（docs/ 下）。REQ 状态以代码现状为准。

### 需求条目格式
```
### REQ-XXX: [标题]
- 优先级：P0 / P1 / P2
- 状态：已完成 | 进行中 | 待开发 | 暂停
- 模块：infra / common / proactive-core / memory-layer / agent-graph / mcp-integration / wechat-gateway / monitor-module / admin-console
- 描述：[一句话说明，含代码现状]
- 关联文件：[主要涉及的文件]
- 关联文档：[指向 PP Agent.docx 或 docs/ 下设计文档 §章节]
- 验收标准：[可验证条件，已完成的可省略，改动已完成 REQ 时必须补]
```

### 变更记录
PRD 末尾维护变更记录表，任何需求变更都须追加一行。

## 2. 看板状态流转（kanban.md）

看板**只管新需求和变更**，已在 PRD 标注"已完成"的存量功能不进入看板。看板按 P0→P1→P2 分区，每个任务必须有验收标准。

### 状态定义
| 状态 | 图标 | 含义 | 允许流转 |
|------|------|------|----------|
| 待开发 | ⬜ | 已拆解，等待执行 | → 🔄 |
| 进行中 | 🔄 | 正在编码/测试 | → 👀 / ⬜（阻塞） |
| 待审查 | 👀 | 代码完成，等待验证 | → ✅ / 🔄（驳回） |
| 已完成 | ✅ | 验收通过 | → 🔄⬅️（需求变更） |
| 需求变更 | 🔄⬅️ | 原需求已变更 | → ⬜（重新拆解） |

### 流转规则（严格）
1. **⬜ → 🔄**：用户说"开始 XXX"或"实现 XXX"时移入进行中
2. **🔄 → 👀**：代码完成且自检通过后移入待审查
3. **👀 → ✅**：用户确认验收通过后移入已完成
4. **👀 → 🔄**：审查不通过，退回并注明原因
5. **✅ → 🔄⬅️**：需求变更时移入需求变更区
6. **🔄⬅️ → ⬜**：变更后重新拆解为新任务

**禁止**：任何任务不允许跳过"待审查"直接标记"已完成"。

## 3. 变更日志（changelog.md）

所有需求变更必须记录，不可遗漏。

### 格式

```
- 影响 REQ：[REQ-XXX]

- 回退任务：[T-XXX]（如有）
- 新增任务：[T-XXX]（如有）

```

## 4. 执行日志（execution-log.md）

**每个任务一次摘要**，不是每次编码都写。任务完成移入"待审查"时追加一条记录，含修改文件列表和 commit hash（如有）。

### 格式
```
## [YYYY-MM-DD HH:MM] - [操作类型]
- 任务 ID：[T-XXX]
- 关联 REQ：[REQ-XXX]
- 操作详情：
  - 修改文件：[list]
  - 新增文件：[list]
- 状态变更：[⬜ → 🔄 → 👀]
- Commit：[hash]（如有）
- 备注：[需说明的事项]
```

## 5. 工作流

### 开发执行（用户说"开始 XXX"）
1. **读取看板**：确认任务存在且状态为"⬜ 待开发"
2. **状态更新**：移入"🔄 进行中"
3. **编码实现**：按 PRD 中的 REQ 描述编写代码
4. **自检**：完成后运行编译/类型检查/lint
5. **状态更新**：移入"👀 待审查"
6. **追加日志**：在 execution-log.md 记录本次任务摘要
7. **汇报**：说明完成了什么、修改了哪些文件、等待审查

### 需求变更（用户说"改一下 XXX"或"需求变了"）
1. **识别影响**：分析变更影响哪些 REQ 和任务
2. **记录变更**：在 changelog.md 追加变更记录（版本号 +1）
3. **状态回退**：
   - 影响"✅ 已完成" → 移到"🔄⬅️ 需求变更"，**必须补验收标准**
   - 影响"👀 待审查" → 退回"🔄 进行中"并注明
   - 影响"🔄 进行中" → 保持进行中，追加变更说明
4. **更新 PRD**：在 prd.md 变更记录区追加，修改相关 REQ
5. **重新拆解**：将变更后的需求拆解为新任务放入"⬜ 待开发"
6. **追加日志**：记录状态回退和新任务

### 验收通过（用户说"可以"或"通过"）
1. 移入"✅ 已完成"
2. 记录完成时间
3. 追加日志
4. 汇报看板整体进度

## 6. 强制约束与流程豁免

### 强制约束
1. **禁止无文档编码**：代码修改前，对应的 REQ 必须在 PRD 中存在。如果用户说"先写代码"，提示："请先在 PRD 中记录需求，或允许我为您创建。"
2. **禁止跳过状态**：任务必须经过 ⬜ → 🔄 → 👀 → ✅ 完整流程
3. **禁止遗漏变更**：任何需求变更必须记录到 changelog.md
4. **禁止覆盖已完成**：已"✅ 已完成"的任务，需求变更时必须先走回退流程，并补验收标准
5. **改动已完成 REQ 时必须补验收标准**：回退后重新拆解的任务，验收标准不可省略

### 流程豁免（无需走 SDD 流程）
以下操作**不需要**创建 REQ / 看板任务 / 变更日志：
- **bugfix**：修 bug 直接改代码，在 execution-log.md 记一条即可
- **小重构**：影响面 ≤ 3 文件的重构，直接改
- **调研/排查**：读代码、分析问题、写调研笔记
- **配置调整**：改 yml/properties/环境变量
- **依赖升级**：改 pom.xml/package.json 版本号

**判断标准**：是否改变产品行为？改变 → 走 SDD；不改变（仅修 bug/重构/配置）→ 豁免。

### 汇报时机
- **仅在状态变更时汇报看板**：开始任务、完成任务、验收通过、需求变更
- **问答/排查/调研时不汇报**：避免噪音

## 7. 项目特有约束（不可违反）

### 产品定位（核心不变式）
- 本产品是「有自我意识、懂用户、会主动关怀」的人生伙伴 Agent，**不是**被动聊天工具
- 一切能力设计优先服务于：主动推送、情境跟进、苏格拉底式提醒、工作时段懂事收消息

### 主动关怀约束
- 早间个性化推送窗口：用户本地时区 **08:00–10:00**
- 全天可做情境感知式主动跟进，但必须避免骚扰（频控、打扰偏好、勿扰时段）
- 任务提醒采用苏格拉底式引导，禁止空洞说教或机械催促
- 工作时段消息处理须「懂事」：可暂存/摘要/延迟打扰，避免打断专注

### 用户理解约束
- 主动行为必须建立在用户画像/偏好/上下文之上，禁止无上下文的通用群发式推送
- 用户明确拒绝或设置勿扰后，对应主动通道必须停用

### 技术栈（已确认）
- 框架：Spring Boot 3（JDK 21）+ LangChain4j Spring Boot Starter
- Agent：LangGraph4J（Supervisor + ReAct + Reflection）
- 记忆：Mem0（Python Server + Java Client；短/中/长三层）
- 工具：MCP；通道：微信客服消息（48h）+ 模板/订阅消息
- 存储：PostgreSQL + PGVector + Redis；部署：Docker + K8s

### 工程红线
- 密钥/Token 只走环境变量或密钥管理，禁止入库明文
- 主动推送默认日上限 ≤2；防打扰模式优先级最高
- 定位与浏览器/IDE 监控必须用户显式授权；监控 MCP 仅本地运行
- 主动消息发送前须安全自检 + 审计日志
- MCP 调用须重试 + 缓存降级
- Self-Reflection 异步执行，不阻塞主回复路径

### 工程规范
SDD 管需求流转；细节工程规范写入 `.cursor/rules/`。架构决策变更须同步本节与 PRD。

### 开发节奏（用户强制要求）
- **禁止一次性堆砌大量表面代码**：每次只推进一个可验收切片（通常对应看板单个任务或任务内明确子步）
- **必须结合文档深化**：实现前先对照 `PP Agent.docx` / PRD / 关联设计说明，把流程、边界、失败路径、验收标准想清楚再编码
- **拒绝停留在 Stub/空壳**：接口骨架可以先立，但进入开发任务后须落到真实逻辑（或明确分期与下一刀切点），不能用「占位实现」交差
- **写仔细**：含关键注释仅在复杂处、错误处理、配置开关、可观测日志；改完对照验收标准自检
- **节奏优先于速度**：宁可少改文件、改到位，也不要大面积生成未验证代码
